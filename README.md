## 학습 방법

- 1주차: 차주 금요일 (4월 10일) 까지 학습한 내용을 바탕으로 테이블 초안 작성하고 간이 성능 테스트를 진행하기
- 2~3주차: 초안 테이블 디벨롭하고 이를 바탕으로 스키마 작성 및 성능 테스트 진행하기
- 4주차:
    - 2, 3주차동안 작성한 테이블 및 스키마를 바탕으로 AI에게 조회하는 쿼리 요청하기
    - AI가 작성해준 쿼리문으로 DB 조회하기
        - 응답 속도 확인하기
        - 부하 테스트
    - Optional
        - 테이블 최적화 시도

- ERD (Entity Relationship Diagram)

    - 2주차 1차 개선 테이블을 이용한 성능 보고서
    - 3주차 2차 개선 테이블을 이용한 성능 보고서

- Optional
    - 쿼리 최적화 시도 및 결과 보고서

## OKR

1. 최종 결과물인 ERD와 DB 테이블 스키마를 이용하여 DBMS를 구축

    - 더 나은 구조가 있는가?

- 브랜치 전략 선정 이유
    - 팀원의 숙련도를 고려해 기초적인 GitHub2 Flow를 기반으로 하되, 실무 효율을 위해 브랜치 구조를 최소화한 맞춤형 전략을 채택함.
    - 프로젝트 진행에 불필요한 `release` 및 `hotfix` 브랜치를 제거한 `main`, `develop`, `feature` 브랜치만으로 구성함.
        - `release` 브랜치: 운영할 목적이 아니므로 삭제
        - `hotfix` 브랜치: 학습 단계에서는 불필요. 추후 서버를 이용한 검증 단계에서 필요할 경우 생성하여 사용.
- 브랜치 네이밍 규칙: 타입 / 기능대상
    - `main`
        - `develop`
            - `feat/A`
            - `feat/B`
            - `feat/N`
